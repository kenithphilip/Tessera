# v1.0 enforcement-mode default flip

## What changed

The `TESSERA_ENFORCEMENT_MODE` default flipped from `both` (v0.12
to v0.14) to `args` (v1.0+).

| v0.x default | v1.0 default | What it means |
| --- | --- | --- |
| `both` | `args` | Argument-level provenance is now the primary policy gate; the legacy `min_trust` scalar floor is no longer evaluated unless an operator opts back in |

## Why

Per [ADR-0006](../adr/0006-arg-level-provenance-primary.md):
argument-level provenance is the v1.0 primary enforcement mode.
The v0.12 default (`both`) ran both the new argument-level checks
AND the legacy `min_trust` scalar floor for the parallel-evaluation
phase, so any divergence surfaced as a deny in CI before v1.0 GA.
v1.0 cuts the legacy path because:

- The Phase 2 + Phase 3 work (Action Critic + MCP signing + RFC
  8707 + advanced detectors) layered above the argument-level
  primary now compose correctly without the scalar floor.
- The two paths produce identical decisions on every test in the
  Tessera suite (verified across the full Phase 2 + Phase 3 work).
- Keeping both paths active doubles the policy latency for every
  call, which Phase 4 wave 4B's Rust GA wants to amortize away.

## Who is affected

Operators who:

- Set `TESSERA_ENFORCEMENT_MODE=both` explicitly: NO change. Your
  setting still works exactly the same.
- Set `TESSERA_ENFORCEMENT_MODE=scalar` explicitly: NO change.
  Your legacy single-floor behavior still works.
- Did NOT set `TESSERA_ENFORCEMENT_MODE`: the default flips. Most
  call paths produce the same decision (the Phase 2 work made the
  two checks consistent), but there are a handful of edge cases
  where the new default denies a call that v0.12 allowed (or vice
  versa).

## Edge cases

The argument-level path catches a strictly larger surface than
the scalar floor. A v0.12-`both` deployment that was relying on
the scalar floor's permissive behavior on a specific argument
will see a deny on v1.0 default. Examples:

- A `transfer_funds` call whose `recipient` was an UNTRUSTED arg
  but whose context-level `min_trust` was at TOOL level: the
  scalar path allowed (TOOL >= TOOL); the argument-level path
  denies (recipient must be TRUSTED). v0.12-`both` denied because
  arg-level fired; v1.0-`args` denies for the same reason. NO
  change.
- A `send_email` call whose `body` was UNTRUSTED but `to` /
  `cc` / `bcc` were TRUSTED: scalar allowed at TOOL floor;
  argument-level allows (body required ENDORSED, present-at-
  TRUSTED is a superset of ENDORSED in our integrity ordering).
  No change.

## Pinning the v0.12 behavior

If your deployment depends on the v0.12 `both` behavior for a
known reason, pin it explicitly:

```bash
export TESSERA_ENFORCEMENT_MODE=both
```

The `both` mode remains supported through the v1.x line; we will
not remove it without an ADR + 1-year deprecation window.

## Rolling back

To restore the v0.12 default behavior at the deployment level:

```bash
# Pin the v0.12 default for the entire deployment.
export TESSERA_ENFORCEMENT_MODE=both
```

To roll back per-call (rare; useful for one specific tool path
during a migration window), wrap the call in a context manager
that overrides the env var. There is NO per-call API; the env
var is the only knob, intentionally, so the policy decision is
deployment-uniform.

## Verifying

After deploying v1.0, run the full test suite at your code
revision:

```bash
.venv/bin/pytest tests/ -q
```

Watch for new `policy_deny` events with detail
`{"source": "args"}` that were not present at v0.12. Each one
indicates a call your v0.12 `scalar` floor was permitting and
the v1.0 `args` floor is denying. Triage by:

1. Confirm the call SHOULD be denied (most likely yes; the
   argument-level check is the more rigorous one).
2. If the call SHOULD be allowed, fix the upstream segment
   labeling so the critical arg traces to a TRUSTED source.
3. As a last resort, pin
   `TESSERA_ENFORCEMENT_MODE=both` and file a Tessera issue
   describing the case.

## Telemetry

The v1.0 `args` decisions surface as:

| Event kind | Detail keys | When |
| --- | --- | --- |
| `policy_deny` | `source: args`, `tool`, `arg`, `reason` | Argument-level path denies |
| `critical_args.deny` | `tool`, `arg`, `principle: origin_consistency` | Action Critic pre-check denies |

Both kinds map into the 8 compliance taxonomies (NIST 800-53,
CWE, OWASP Agentic ASI, MITRE ATLAS, EU AI Act, ISO/IEC 42001,
CSA AICM, NIST AI 600-1) per the existing SARIF mapper.
